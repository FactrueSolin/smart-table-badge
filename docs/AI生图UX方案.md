配套文档：

- [`AI生图架构.md`](./AI生图架构.md)
- [`AI生图数据库设计.md`](./AI生图数据库设计.md)

## 1. 文档目标

基于现有 AI 生图技术方案，为管理后台设计一套可直接落地的 UX 方案，确保以下目标同时成立：

- 管理员能快速创建文生图任务
- 异步任务过程可感知、可追踪、可恢复
- 生成结果能自然回收到现有图库资产链路
- 信息架构不膨胀，不把 AI 生图做成脱离当前后台的新系统

本方案默认第一阶段仅支持文生图 `text_to_image`，不开放图生图入口。

## 2. 设计范围与约束

### 2.1 范围

- 管理后台中的 AI 生图入口设计
- 创建任务表单
- 任务列表与任务详情
- 结果查看、导入后使用、失败恢复
- 与现有图床管理的协同方式

### 2.2 约束

- 后端采用任务制异步流程，不提供前端同步阻塞等待出图
- 第一阶段仅一个 Provider，但前端不得暴露 Provider 心智
- 结果图片最终进入本地图片资产库，不直接依赖第三方临时 URL
- 状态反馈需要同时兼容 SSE 推送和手动刷新
- 当前后台已有一级导航，AI 生图不应新增顶层入口

## 3. 用户角色与核心场景

### 3.1 目标用户

- 运营人员：快速产出活动图、背景图、KV 图
- 内容编辑：生成可直接用于页面展示的图片素材
- 管理员：查看任务状态、排查失败、管理生成结果

### 3.2 核心任务

1. 输入提示词并发起生成
2. 在不离开页面的情况下确认任务是否成功
3. 直接使用生成结果的原图链接或展示页链接
4. 失败时知道失败原因，并能快速重新创建任务

## 4. UX 设计原则

### 4.1 先完成闭环，再做高级能力

第一阶段优先保证“提示词 -> 任务 -> 结果入库 -> 使用素材”的闭环顺畅，不为了参数丰富度牺牲理解成本。

### 4.2 异步必须有安全感

用户不应因为页面刷新、切换标签、网络抖动而怀疑任务丢失。界面需要持续表达三件事：

- 任务已成功提交
- 系统仍在处理中
- 结果会自动进入图库

### 4.3 结果优先，不强调底层实现

前端主语言围绕“生成任务”“结果图片”“导入图库”，不暴露 Provider 术语、远端任务细节或技术状态噪音。

### 4.4 低门槛创建，高密度排查

- 创建任务时只展示必要字段
- 排查失败时提供完整状态、时间线、错误信息和参数快照

## 5. 信息架构

### 5.1 一级入口

将 AI 生图放入现有后台的 [`图床管理`](/var/tmp/vibe-kanban/worktrees/2954-ux/htmlpush/app/admin/page.tsx) 内，不新增顶层导航。

原因：

- AI 生图的最终产物是图片资产
- 当前后台已存在“上传图片并生成页面”的心智
- 可避免出现“图库”和“AI 图片任务”割裂管理

### 5.2 二级结构

在 `图床管理` 内增加二级切换：

- `图片资产`
- `AI 生图`

建议默认仍停留在 `图片资产`，避免影响现有使用习惯。

### 5.3 关键页面对象

- AI 生图工作台
- 任务详情抽屉
- 图片资产列表中的 AI 结果标识

## 6. 页面结构方案

### 6.1 AI 生图工作台

建议采用桌面端双栏、移动端单栏的布局：

- 左侧：创建任务卡片
- 右侧：最近任务列表

桌面端推荐比例：`5 / 7`

这样可以让“输入”和“结果反馈”同时可见，减少用户在表单与任务详情之间来回跳转。

示意结构：

```text
图床管理 / AI 生图
----------------------------------------------------------
| 创建任务                                   | 最近任务   |
| 名称                                       | 状态筛选   |
| Prompt                                     | ---------- |
| Negative Prompt（折叠）                    | 任务卡片 A |
| 模型 / 尺寸                                | 任务卡片 B |
| 高级参数（折叠）                           | 任务卡片 C |
| [立即生成]                                 |            |
----------------------------------------------------------
```

### 6.2 创建任务卡片

#### 必填字段

- `任务名称`
- `Prompt`
- `模型`
- `尺寸`

#### 选填字段

- `Negative Prompt`
- `Seed`
- `Steps`
- `Guidance`

#### 交互要求

- 默认只展开基础字段
- 高级参数默认折叠
- 如果仅有一个模型可用，显示为只读说明，不做下拉选择
- `任务名称`支持自动预填
  - 默认取 Prompt 前 12 到 20 个字符
  - 用户修改后不再自动覆盖
- `Prompt` 输入区使用多行文本框，支持 `Cmd/Ctrl + Enter` 提交
- 提交后按钮进入 loading 态，并禁止重复点击

#### 表单辅助文案

- Prompt 提示：`描述画面主体、风格、构图和光线，越具体越稳定`
- Negative Prompt 提示：`用于排除低质、模糊、水印等问题，可留空`
- Seed 提示：`留空表示随机生成`

### 6.3 最近任务列表

任务列表按 `createdAt desc` 展示，默认展示最近 20 条。

每条任务卡片展示：

- 任务名称
- 当前状态
- 创建时间
- Prompt 摘要
- 模型与尺寸
- 输出数量摘要
- 主要操作按钮

推荐操作：

- `查看详情`
- `刷新状态`
- `取消任务`
- `复制参数再生成`

其中：

- `取消任务` 只对非终态任务显示
- `刷新状态` 本质触发一次详情同步，不要求用户理解 `sync=true`
- `复制参数再生成` 用于失败恢复和结果迭代

### 6.4 任务详情抽屉

点击任务卡片或 `查看详情` 后，从右侧打开详情抽屉，不离开当前工作台。

详情抽屉包含四个模块：

1. 基本信息
2. 状态时间线
3. 输出结果
4. 参数快照

#### 基本信息

展示：

- 任务名称
- 当前状态
- 创建时间、提交时间、完成时间
- 模型、尺寸
- 错误摘要

#### 状态时间线

按时间顺序展示：

- 已创建
- 已提交
- 处理中
- 已完成 / 已失败 / 已取消 / 导入失败 / 已超时

如果后端提供 `syncAttempts` 和 `lastSyncedAt`，应在时间线底部展示为辅助信息，不做主视觉焦点。

#### 输出结果

成功任务至少展示：

- 结果缩略图
- 图片资产 ID
- 原图链接
- 展示页链接
- `复制直链`
- `打开展示页`

如果有多张结果图，按 `outputIndex` 顺序平铺显示。

#### 参数快照

展示本次任务的参数，只展示业务字段：

- Prompt
- Negative Prompt
- 模型
- 尺寸
- Seed
- Steps
- Guidance

不展示 Provider 原始请求和原始响应 JSON。

### 6.5 图片资产列表中的 AI 结果表达

在现有 `图片资产` 列表中，为 AI 生成图片增加来源标识：

- `AI 生成`
- `手动上传`

对于 AI 生成图片，建议在列表或详情中补充：

- 来源任务名称
- 生成时间
- Prompt 摘要

这样用户在图库里能反向追溯生成来源，而不是把 AI 结果当作普通上传图处理。

## 7. 核心用户流

### 7.1 创建任务

1. 用户进入 `图床管理 > AI 生图`
2. 输入 Prompt，补充名称和基础参数
3. 点击 `立即生成`
4. 表单进入提交中状态
5. 创建成功后：
   - 表单顶部出现成功反馈：`任务已提交，正在生成中`
   - 任务列表顶部插入新任务
   - 自动高亮该任务 3 秒
   - 默认不清空 Prompt，方便继续微调生成

### 7.2 查看进度

1. 任务创建后进入 `submitted` 或 `processing`
2. 列表卡片持续显示进行中状态
3. 页面通过 SSE 接收状态更新
4. SSE 不可用时，用户仍可点击 `刷新状态`
5. 当任务完成时：
   - 卡片状态更新为成功
   - 结果缩略图出现在卡片或详情抽屉中
   - 顶部出现轻提示：`图片已生成并导入图库`

### 7.3 使用结果

成功后用户可直接执行：

- 复制原图直链
- 打开展示页
- 前往 `图片资产` 查看全部结果
- 复制参数再次生成相似版本

建议在成功态提供一个捷径按钮：`在图库中查看`

### 7.4 失败恢复

不同失败类型采用不同反馈策略：

- 提交失败：表单内联报错，允许用户直接修改后重试
- 处理中失败：任务卡片显示失败标签和简短原因
- 导入失败：明确提示“图片已生成，但导入图库失败”
- 超时：提示“生成耗时过长，系统已停止等待”

失败态首选操作：

- `查看详情`
- `复制参数再生成`

第一阶段不建议在前端暴露“重新导入”之类后端尚未承诺的操作。

## 8. 状态映射与界面文案

| 后端状态 | 前端标签 | 用户可理解说明 | 推荐视觉 |
|------|------|------|------|
| `queued` | 排队中 | 任务已创建，等待提交 | 中性灰 |
| `submitted` | 已提交 | 已发送到生成服务 | 蓝色 |
| `processing` | 生成中 | 正在生成图片 | 蓝色动效 |
| `succeeded` | 已完成 | 图片已生成并导入图库 | 绿色 |
| `failed` | 生成失败 | 服务返回失败，请调整参数后重试 | 红色 |
| `timed_out` | 已超时 | 处理时间过长，系统已停止等待 | 橙色 |
| `canceled` | 已取消 | 任务已取消，不再继续处理 | 灰色 |
| `import_failed` | 导入失败 | 图片已生成，但未成功导入图库 | 橙红色 |

文案要求：

- 不显示英文状态码
- 不显示 `Provider`、`remoteTaskId` 等底层术语
- 失败原因优先用可读中文，其次才展示脱敏后的技术信息

## 9. 空状态、加载态、异常态

### 9.1 空状态

AI 生图页首次进入且没有任务时，展示：

- 标题：`还没有生成任务`
- 描述：`输入提示词后，系统会异步生成图片并自动收录到图库`
- 辅助示例 Prompt

### 9.2 加载态

- 首屏加载任务列表时使用骨架屏
- 创建任务按钮在提交中显示 `提交中...`
- `processing` 卡片使用轻量动效，不要整卡闪烁

### 9.3 SSE 断连

不需要打断用户操作，但要给非阻塞反馈：

- 顶部轻提示：`实时更新已断开，可手动刷新任务状态`

### 9.4 列表查询失败

展示局部错误块，不影响创建表单：

- 文案：`任务列表加载失败`
- 操作：`重新加载`

## 10. 表单与交互细节规范

### 10.1 防重复提交

- 点击 `立即生成` 后按钮禁用
- 前端应生成一次请求级幂等键，避免双击造成重复任务

### 10.2 参数校验

前端只做轻校验：

- 必填项非空
- 字数上限提示
- 数值字段格式合法

模型白名单、尺寸白名单、步数范围等裁决以后端为准。前端只负责把错误翻译成可读提示。

### 10.3 任务取消

- 点击 `取消任务` 前需二次确认
- 确认文案：`取消后任务不会继续推进，已生成结果也不会再导入`

### 10.4 结果复制

复制成功后使用局部反馈：

- `已复制直链`
- `已复制展示页链接`

不要复用全局唯一的复制状态，避免多个任务卡片之间互相污染。

## 11. 与现有后台视觉风格的衔接

当前后台已经使用：

- 卡片式分组
- 圆角按钮与轻量边框
- 表格和列表混合的信息展示

AI 生图区域应延续现有风格，不单独引入全新设计语言。重点变化应体现在信息组织，而不是视觉跳变。

建议延续现有模式：

- 卡片容器承载一个明确任务
- 状态使用标签而非大面积提示框
- 详情用抽屉，避免整页跳转

## 12. 第一阶段不做的 UX 能力

- 不开放图生图入口
- 不开放批量生图
- 不开放多 Provider 切换
- 不开放风格市场、Prompt 模板市场
- 不开放任务级协作、评论、收藏

这些能力会显著抬高复杂度，不适合作为第一阶段入口体验的一部分。

## 13. 设计验收标准

满足以下条件即可认为 UX 方案达到可实现状态：

1. 用户可在 10 秒内找到 AI 生图入口并创建首个任务
2. 用户在任务异步处理中不会误以为任务丢失
3. 用户完成出图后，可在 1 次点击内拿到原图链接或展示页
4. 用户在失败时能知道下一步动作，而不是只看到报错
5. AI 生成结果与手动上传图片能在同一图库中统一管理

## 14. 后续演进建议

如果第一阶段稳定运行，再考虑第二阶段增强：

- 图生图模式接入
- Prompt 历史与快捷复用
- 任务筛选：仅看失败、仅看今日、仅看某模型
- 结果多选入库操作
- 与代码编辑器联动，把图库图片快速插入 HTML
